## Launch Blockers

Post-review update: items 1-4 were addressed in the Core Marketplace Correctness pass on 2026-07-19. Item 5 remains open for the next seller/Ops governance phase.

1. Payment method validation is incomplete.
   - Location: `services/api/src/modules/customer.ts:270-316`
   - Checkout accepts any `paymentMethod` string and only treats literal `cod` specially.
   - Risk: disabled/fake payment methods can create orders, and online orders enter `PENDING` without a provider order.
   - Fix status: checkout now validates against Admin-enabled payment config. Real gateway order creation remains a production integration gap.

2. Seller order state transitions are too permissive.
   - Location: `services/api/src/modules/seller.ts:189-207`
   - Current API can move orders across lanes without enforcing valid previous state.
   - Risk: seller can hand over or deliver before confirm/invoice/bag pack, and timeline/customer SLA can become wrong.
   - Fix status: strict transition state machine and tests added.

3. Parent order status and customer rewards are not fully reconciled.
   - Location: `services/api/src/modules/customer.ts:311-348`, `services/api/src/modules/seller.ts:189-223`
   - Parent order remains independent from sub-order progress; rewards are documented but not awarded on completed checkout/delivery.
   - Risk: customer sees stale parent order state and rewards feature does not match product promise.
   - Fix status: parent rollup rules and reward posting ledger added. Reward reversal policy remains open.

4. Admin and Ops product approval paths are inconsistent.
   - Admin path: `services/api/src/modules/admin.ts:130-145`
   - Ops path: `services/api/src/modules/ops.ts:59-109`
   - Admin approval creates product without the same image/default legal fields/seller notification behavior that Ops approval uses.
   - Risk: product approved from one screen behaves differently from the other.
   - Fix status: shared approval/rejection service added for Admin and Ops.

5. Role-wise access is still coarse for Support/Admin.
   - Admin router allows `ADMIN` and `SUPPORT` broadly, with only a few admin-only routes.
   - Risk: support users may perform actions that should belong to finance/catalogue/admin only.
   - Fix: implement permission matrix for Admin, Support, Logistics, Finance, Catalogue Ops, and Super Admin.

## Important Gaps Before Beta

- Google Maps Places/address autocomplete is not yet connected; lat/lng is stored but user capture is still prototype-level.
- Store-live gating does not fully enforce mandatory approved documents before seller becomes live.
- Product/document uploads need object storage, signed upload URLs, attachment scanning, and admin preview.
- Refund flow needs payment gateway refund creation, webhook reconciliation, and customer notification.
- COD flow needs reconciliation, remittance status, failed COD handling, and settlement ledger.
- Push notifications need FCM/APNs, delivery logs, retry, read/unread state, and user preferences.
- Seller analytics/reports need API-backed downloadable CSV/PDF exports.
- Invoice/label PDFs need visual regression QA and final legal/compliance review.
- Mobile apps have typecheck only; they need real UI/E2E smoke tests.
- Admin web needs browser E2E tests for login, orders, refunds, catalogue approval, support replies, settings, and audit logs.
